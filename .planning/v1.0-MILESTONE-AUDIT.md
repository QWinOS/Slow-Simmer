## Integration Findings

### 🔴 BLOCKER: Payment Failure Recovery Broken

**Integration gap between Phase 2 and Phase 3**

After a RazorPay payment failure, the "Back to Form" button (PaymentSection.tsx:208) scrolls to `#form` but `RegistrationFlow` still renders `PaymentSection` because `data` remains truthy in context. The `RegistrationProvider` has no `clearRegistrationData` function.

**Impact:** User cannot re-edit registration form after payment failure without page refresh (which loses all context).

**Files involved:**
- `components/RegistrationProvider.tsx:21-24` — no reset function exposed
- `components/PaymentSection.tsx:204-213` — only calls `setStatus("idle")`
- `components/RegistrationFlow.tsx:10` — `if (data)` check prevents form re-render

### ⚠️ PARTIAL: Dual-Write Race Condition

**Between `/api/orders/verify` and webhook handler**

The verify route and webhook both call `checkPaymentIdExists → appendRegistrationRow`. If the verify route's write hasn't completed when the webhook's `checkPaymentIdExists` reads column N, a duplicate row can be written.

### ⚠️ CRITICAL: UI-03 Brand Not Applied

**Theme gap in Phase 1 (verified, not fixed)**

Phase 1 VERIFICATION confirmed two gaps:
1. **Typography wrong:** Playfair Display SC and Karla never applied — body renders Inter, headings render Inter via miswired `font-heading` utility
2. **Palette not applied:** shadcn tokens remain zinc defaults — warm red/gold palette never implemented

These directly contradict UI-03 ("clean, elegant design fitting a supper club brand"). The VERIFICATION flags them as small localized fixes (layout.tsx + globals.css) but they were never remediated.
