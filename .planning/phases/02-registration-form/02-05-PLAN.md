    ## Test Cases (per VALIDATION.md Per-Task Verification Map)

    ### 1. Form renders all sections (FORM-01 through FORM-07)
    - Verify "Reserve Your Spot" heading is rendered
    - Verify all section titles exist: "Personal Information", "Guest Details", "Social & About"
    - Verify all field labels exist: "Full Name", "Contact Number", "Email Address", "Aadhar Number", "Bringing a guest?", "About Yourself", "Instagram or LinkedIn"
    - Verify submit button renders with text "Submit Registration"

    ### 2. Required fields show asterisk (D-11)
    - Verify "Full Name *", "Contact Number *", "Email Address *", "Aadhar Number *" labels contain asterisk spans
    - Verify "About Yourself" and "Instagram or LinkedIn" do NOT have asterisks

    ### 3. onBlur validation shows inline errors (D-07, FORM-08)
    - Type single character in name field, tab away → expect "Name must be at least 2 characters" error
    - Type invalid contact (e.g. "12345"), tab away → expect validation error
    - Type invalid email (e.g. "notanemail"), tab away → expect validation error
    - Note: Aadhar formatting makes blur testing complex — focus on name/contact/email for blur tests
    - Verify FieldError component renders with error message

    ### 4. Guest fields toggle with checkbox (D-05, FORM-04)
    - Initially: Guest Name and Guest Age inputs should NOT be visible
    - Click "Bringing a guest?" checkbox
    - After click: Guest Name and Guest Age inputs should be visible (presence in DOM or not hidden)
    - Uncheck: fields should hide again

    ### 5. Error summary banner appears on submit with errors (D-09, D-19, FORM-08)
    - Click submit without filling any fields
    - Verify error summary with `role="alert"` is rendered
    - Verify error summary contains "Please fix the following errors:" heading
    - Verify error messages are listed (multiple error entries)

    ### 6. Submit succeeds with valid data (D-15, D-18)
    - Fill all required fields with valid data:
      - name: "Test User"
      - contact: "9876543210"
      - email: "test@example.com"
      - aadhar: type "123456789012" (12 digits)
    - Click submit
    - Verify no error summary is rendered
    - Note: toast and scroll behavior are side effects of the submit handler. The toast fires from sonner — testing toast appearance requires mocking sonner or checking that the submit handler completes without errors. For this test, verify the form submits without rendering errors.

    ### 7. Aadhar input masks and formats (D-12, D-13, FORM-07)
    - Type "123456789012" into Aadhar field
    - Verify the displayed value is formatted with spaces (the input value includes spaces via formatAadhar)
    - Verify the field type is "password" (masked)
    - Note: because formatting happens client-side and the Controller manages value, DOM assertions need to access the underlying input value. Use `screen.getByLabelText(/aadhar/i)` and check `getAttribute("type")`.

    ### 8. Guest fields conditionally required (D-06)
    - Check "Bringing a guest?"
    - Fill guest name and age with empty/whitespace
    - Submit → verify guest-related errors appear
    - Fill valid guest name and age
    - Submit → no guest errors

    ## Mocking strategy

    Use `vi.mock()` if needed to silence sonner's toast (which needs a DOM portal):
    ```typescript
    vi.mock("sonner", () => ({
      toast: {
        success: vi.fn(),
        error: vi.fn(),
      },
    }))
    ```
    This prevents test failures from Toaster not being rendered in the test DOM.

    ## Running tests

    Run `npx vitest run tests/RegistrationForm.test.tsx --reporter=verbose` to verify all tests pass.

    ## Important constraints

    - Use `userEvent.setup()` pattern (async) instead of fireEvent — userEvent simulates realistic interactions (blur events, focus, keyboard)
    - Wrap async operations in `await` with `userEvent`
    - Use `screen.getByRole`, `screen.getByLabelText`, `screen.getByText` for queries (not test IDs unless necessary)
    - Do NOT test sonner toast functionality — it's a third-party library. Test that the submit handler is called.
    - Do NOT test scroll behavior — `document.getElementById("payment")` won't exist in test DOM. Test that submit doesn't throw.
    - Keep tests focused on user-observable behavior, not internal implementation
