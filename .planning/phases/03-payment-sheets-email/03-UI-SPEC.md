## State Machine

The `PaymentSection` component uses a 4-value state machine via `useState<PaymentStatus>`:

```
idle ──(data is null)──► placeholder (show muted text)
  │
  └──(data exists)──► summary card
  │
  └──(user clicks Pay)──► awaiting
       │
       ├──(verify succeeds)──► success
       ├──(verify fails)─────► failure
       ├──(payment.failed)───► failure
       ├──(order/create 500)─► failure
       ├──(fetch error)──────► failure
       └──(modal dismissed)──► idle (back to summary)
```

**Status values:**
- `"idle"` — Default state. Shows either placeholder (if `data` is null) or summary card (if `data` exists).
- `"awaiting"` — RazorPay modal is open or payment verification in progress. Button is disabled, shows spinner.
- `"success"` — Payment verified. Shows green success card.
- `"failure"` — Payment failed or cancelled. Shows red failure card with back-to-form link.

## Loading, Empty & Error States

### PaymentSection State Table

| State | Condition | Visual | Duration |
|-------|-----------|--------|----------|
| Placeholder | `data` is null | Muted text in CardContent: "Fill the registration form above to proceed to payment." | Until form is submitted |
| Summary Card | `data` exists, status is `"idle"` | Card with event details summary + CTA button | Until user clicks Pay or form is resubmitted |
| Awaiting | status is `"awaiting"` | Button disabled with `Loader2` spinner + "Processing your payment..." | While RazorPay modal is open / payment is being verified |
| Success | status is `"success"` | Green card with checkmark, confirmation text | Persistent |
| Failure | status is `"failure"` | Red card with error text + back-to-form button | Until user clicks "Back to Form" (resets to idle) |

### Error Recovery Paths

| Failure Point | UX | Recovery |
|--------------|-----|----------|
| Order creation API fails (500) | Status → "failure", failure card shown | User clicks "Back to Form" → re-fills form → retries |
| Network error (fetch throws) | Status → "failure", failure card shown | User clicks "Back to Form" → retries |
| Signature verification fails | `POST /api/orders/verify` returns non-ok → status → "failure" | User retries via Back to Form |
| RazorPay modal payment failure | Modal `payment.failed` event → status → "failure" | User retries via Back to Form |
| User closes modal without paying | Modal `ondismiss` → status → "idle" | Summary card shown again, user can retry |
| RazorPay script fails to load | `loadRazorpayScript()` rejects → catch block → status → "failure" | User retries via Back to Form |

## Responsive & Accessibility Requirements

### Mobile Adaptations

- **Touch targets:** Pay button must have minimum 44×44px touch target (uses shadcn button default sizing)
- **Card width:** `max-w-lg` (512px max), full-width on mobile (`px-4` section padding)
- **Card padding:** Shadcn card default (responsive via shadcn)
- **Scrolling:** After form submit, auto-scroll to `#payment` (already implemented in Phase 2 form via RegistrationForm's onSubmit)
- **Back link:** "Back to Form" scrolls smoothly to `#form` section via `document.getElementById("form")?.scrollIntoView({ behavior: "smooth" })`

### Accessibility

| Requirement | Implementation Status |
|-------------|----------------------|
| Status announcements | **GAP:** No `aria-live="polite"` region on the payment section container. Screen readers won't announce success/failure transitions. Should be added to the section wrapper. |
| Success icon | `aria-hidden="true"` on `CheckCircle2` — PASS |
| Failure icon | `aria-hidden="true"` on `XCircle` — PASS |
| Button label | "Pay ₹{dynamic} via RazorPay" — descriptive for screen readers — PASS |
| Awaiting spinner | Uses `Loader2` with `animate-spin`; no explicit `aria-label` or `role="status"` on the icon — **GAP** should use `role="status"` |
| Back link | `Button variant="outline"` with inline onClick — role is implicit from button — PASS |
| Color contrast | Green success card: `text-green-700` on `bg-green-50` — verify WCAG AA (green-700 ≈ #15803D on green-50 ≈ #F0FDF4, ~5.5:1 contrast) — PASS |
| Reduced motion | No custom animations in PaymentSection; uses shadcn default transitions only — PASS |
| Section heading hierarchy | `h2` heading "Complete Your Payment" — consistent with other sections (Hero is h1, others are h2) — PASS |

## Copywriting: User Flow

### Success Flow

1. Guest fills registration form → clicks Submit → form validates → `RegistrationProvider.setRegistrationData()` called → auto-scroll to `#payment`
2. Guest sees section heading "Complete Your Payment", summary card with "Review Your Registration" and event details (Event, Date, Time, Guest)
3. Guest clicks "Pay ₹{price} via RazorPay"
4. RazorPay checkout modal opens with pre-filled name, email, contact
5. Guest completes payment in modal (UPI, card, etc.)
6. Modal closes → client calls `POST /api/orders/verify` with payment signature
7. On verification success → PaymentSection shows green card: "Registration confirmed! Check your email for details."
8. Webhook fires server-side → Sheets append + Brevo email sent (background)

### Failure Flow

1. Guest clicks "Pay" → RazorPay modal opens
2. Guest completes/cancels payment → modal closes
3. On failure (any reason): PaymentSection shows red failure card
4. Guest clicks "Back to Form" → smooth scroll to `#form` → can modify details and retry

### No Destructive Confirmations

No delete/clear actions in this phase. The only user decisions are payment (which has RazorPay's own native confirmation) and "Back to Form" (which is navigation, not destructive).
